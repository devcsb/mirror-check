버튼 클릭 후 로컬 저장과 네트워크 호출을 관찰해서 셋 중 하나로 분류합니다.

FAKE

거울을 보고 있었네요

localStorage나 쿠키만 바뀌고 서버엔 아무것도 안 갑니다. 당신 화면에서만 숫자가 올라가는 가짜 카운터.

local network
REAL

서버에 잘 닿았어요

fetch·XHR·beacon 중 하나로 서버에 요청이 갔고, 로컬 저장은 없습니다. 진짜 집계일 가능성이 높습니다.

local network
MIXED

서버 + 중복방지 캐시

서버 요청과 로컬 저장이 모두 발생. 대부분 "이미 눌렀음"을 기억하려는 정상 패턴입니다.

local network

상단 Mirror Check 버튼을 브라우저 북마크바로 끌어다 놓습니다.

의심 페이지로 이동

"도움됐어요", "인기 급상승" 같은 버튼이 있는 페이지로 갑니다.

북마크 → 버튼 클릭

북마크로 감지기를 켠 뒤, 검사하고 싶은 버튼을 누릅니다.

판정 확인

우하단 패널에 FAKE/REAL/MIXED가 즉시 표시됩니다.

how it works

거창한 AI 없이, 세 줄 규칙.

JavaScript의 기본 API를 후킹해 클릭 전후의 부수효과를 비교합니다.

  • 1

    로컬 쓰기 감지

    Storage.prototype.setItem과 document.cookie setter를 monkey-patch해 localStorage·sessionStorage·쿠키 쓰기를 기록합니다.

  • 2

    네트워크 후킹

    fetch, XMLHttpRequest.open, navigator.sendBeacon을 래핑해 모든 네트워크 호출을 가로챕니다.

  • 3

    2.5초 윈도우 비교

    클릭 직후 2.5초 동안 수집된 시그널을 아래 규칙으로 분류해 판정을 내립니다.

limitations

만능은 아닙니다.

아래 경우에는 잘못 판정할 수 있으니 결과를 맹신하지 마세요.

01 / SW + WS

Service Worker · WebSocket

SW가 가로채거나 WebSocket으로 전송하면 fetch·XHR 훅에 안 걸려 REAL인데 FAKE로 오판할 수 있습니다.

02 / batching

디바운스 배치 전송

클릭을 모았다가 수 초 뒤 보내는 사이트는 2.5초 윈도우를 놓칠 수 있습니다. windowMs를 늘려 사용하세요.

03 / CSP

엄격한 CSP

일부 페이지는 Content-Security-Policy로 인라인 스크립트 실행 자체를 막아 북마클릿이 동작하지 않을 수 있습니다.
